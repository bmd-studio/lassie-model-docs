Overview

Classes summary
Account_model

The Account Model handles the money accounts that are linked with persons or organisations. There are currently three types of accounts: first account (person), second account (person) and a general account (organisation).
Api_client_model

The Api Client model describes the external applications linked to Boris where Boris functions as a client
Api_model The API Model class interfaces with data associated with the API Module.
Bar_activity_model The Bar Module can create Bar Activities that are retrieved and created in this class.
Cronjob_model The Cronjob Model the execution of cronjobs.
Currency_model The Currency Model manages all the available types of payments with cash.
Definition_model

The Definition Model handles data that defines a certain set of (semi-)static values. For example: person genders, person languages, person generations, etc.
Development_model The Development Model allows a developer to implement manipulations that might be needed for development.
Event_model The Event Model handles activities where people can subscribe to.
Form_model The Form Model is an interface for the Form Library to retrieve properties of posted forms, such as validation parameters.
Group_model The Group Model manages all the groups people can be in (e.g. committees).
Import_model The Import Model is an interface for the Import Library to get properties of the database structure.
Ion_auth_model Ion Auth Model is an interface for the Ion Auth Library to verify login credentials and store session variables.
Log_model The Log Model allows the framework to quickly insert logs during certain processes.
Manager_model The Manager Model is the interface for the Manager Library that implements a simple Content Management System.
Matchmaker_model The MatchMaker Model is the core of BORIS and handles all the rights given to Persons, Modules and API's.
Membership_model The Membership Model handles memberships people can subscribe to.
Migration_model The Migration Model implements an interface for the Base Migration Library.
Module_model The Module Model describes what modules are registred in this framework and can be handled in the Rights Module.
Ollie_model Ollie Model is an experimental implementation of allowing offline versions of the framework to be exported and used locally. The data can be synced later on after usage.
Owis_model OWIS Model is a specific implementation to interface with OWIS exports provided by the TU/e
Person_model Person Model allows retrieval of personal information and collections of persons in various ways.
Pool_model Pool Model implements the functionality of 'virtual fields'. This enables the front-end user to add fields to existing records without manipulating the database structure.
Rental_model Rental Model handles the very specific functionalities of the Rental Module. Such as, return of items, refunds of bail, etc.
Report_model Report Model allows the user to request meta data about the various Transaction Modules.
Setting_model Setting Model loads all the distribution specific settings and defines them as PHP globals
Syncer_model

The Syncer model is able to identify changed records that are currently unsynced with specific services
Transaction_model

Transaction Model is the interface for the Transaction Library and features the retrieval of all the transaction related records. It can retrieve information about products, transactions, refunds, reports, price queue, product categories and transaction types.
Treasurer_model Treasurer Model is a specific implementation for the Treasurer to handle transaction related manipluations, such as refunds.
Upload_model Upload Model is a verification helper to clean the uploads folder by checking if each file is used in the database.
